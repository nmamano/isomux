# Held-Back Messages Bug Investigation

**Date**: 2026-04-01 (fixed and verified 2026-04-20)
**Status**: Fixed in Isomux via the persistent-consumer architecture described below. Repro verified end-to-end (backgrounded bash + Monitor tool both deliver notifications on their own, no user message needed).
**Tracking issue**: `nmamano/isomux#9`
**Upstream (closed, not a bug)**: `anthropics/claude-agent-sdk-typescript#295`
**Related Isomux commits**: `4b7ba76` (abort-race fix), `9c7600e` (stuck-thinking fix) — these were unrelated abort-path fixes; they did not address this bug.

## The Bug

Bot responses sometimes get "held up" and don't appear in the UI until the next user message is sent. When they do appear, they arrive instantly (proving they were already generated) and appear as a reply to the previous request rather than the current one — an off-by-one response.

### Reproduction

1. Send a message that triggers a `run_in_background` Bash command (e.g., `sleep 5`).
2. Wait for the background task to complete.
3. Send a new message.
4. Observe: the response to the background task completion appears first (instantly), followed later by the response to the new message.

Example from a real conversation:
```
User: "serve the site over 8080 to test"
Agent: [runs python3 -m http.server 8080 in background] "Serving at http://localhost:8080"
  -- background task completes, but no output appears --
User: "explain how the counter gets updated?"
Agent: "Looks like it exited immediately. Let me restart it properly." ← stale response to bg task
  -- later --
Agent: [actual answer about the counter] ← response to user's real question
```

## Root Cause (confirmed)

Isomux uses the V2 SDK API (`unstable_v2_createSession`). Key V2 behaviors (verified with direct repros, see below):

1. `session.stream()` returns an `AsyncGenerator<SDKMessage, void>` that **ends at the turn's `result` message** (`r.done === true`). It is per-turn, not long-lived.
2. Events generated between turns (notably `task_notification` from a backgrounded `Bash`) are **buffered by the SDK** until the next `stream()` call.
3. A fresh `stream()` call does **not** require an intervening `send()`. If there are buffered events it delivers them immediately; if there are no buffered events it **blocks** (async) until an event arrives.
4. If a `stream()` generator is already active (blocked) when `session.send(text)` is called, the response flows through that active generator — you don't need to re-call `stream()` after `send()`.

Isomux violates (2)/(3) by only calling `session.stream()` inside its `sendMessage` path. Between user turns, no consumer is iterating — so any `task_notification` generated by a completed background task sits buffered until the user types the next message. When `sendMessage` starts its new stream iteration, the buffered notification is drained first (before the response to the new message), and the host processes it in order — producing the off-by-one symptom.

## Fix: persistent consumer loop

Run a consumer task for the session's lifetime that keeps calling `session.stream()`:

```ts
while (!closed) {
  for await (const msg of session.stream()) {
    processMessage(agentId, msg);
  }
  // stream ended at this turn's result — loop around; next stream() call blocks until more events arrive
}
```

Implications for `server/agent-manager.ts`:

- Start the consumer when the session is created (`createSession`). Store its promise on the `managed` record so shutdown can await it.
- `sendMessage` stops running its own `for await` loop. It calls `session.send(text)` and waits on a per-turn "result arrived" promise that the consumer resolves when it sees the `result` for this turn. This preserves `sendMessage`'s existing completion semantics (callers await it and know the turn is done).
- Same refactor for `executeSkill`.
- `session.close()` unblocks the consumer's pending `stream()` call (verified in the repro), so shutdown works without extra coordination.

## Evidence (repro scripts)

All four scripts use the V2 API with `pathToClaudeCodeExecutable` pointing to the Isomux-bundled native Claude binary.

- `/tmp/sdk-heldback-repro.ts` — Sends a message that starts a bg `sleep 5`, drains turn 1 until `result`, then waits 20s iterating the now-ended generator. Stream yields nothing (the generator is done; the bug reproduces because nothing re-calls `stream()`).
- `/tmp/sdk-heldback-repro2.ts` — After turn 1 ends and an 8s idle wait (bg task completes during the wait), calls `session.stream()` a second time with **no** intervening `send()`. The buffered `task_notification` is delivered immediately. Proves events are held for the next `stream()` call, not the next `send()`.
- `/tmp/sdk-empty-stream-test.ts` — After turn 1 ends and a 3s idle wait with no bg task (nothing should be buffered), a fresh `session.stream()` blocks for the full 8s timeout without yielding. Proves empty `stream()` does not spin or end-immediately, so the persistent consumer doesn't need polling.
- `/tmp/sdk-persistent-consumer-test.ts` — Starts a persistent consumer iterating `stream()`, then calls `session.send(text)` from the main task while the consumer is parked inside its generator. The consumer's already-active generator receives `system/init`, the `assistant` message (with the response), and the `result`, then ends; loop re-enters for the next turn. Proves send() + persistent consumer compose correctly with no additional coordination.

## Theories that turned out to be wrong

For future-us, so we don't re-introduce them:

1. **"Stream stays open across turns; result doesn't close the generator."** Wrong for V2. `stream()` ends at each turn's `result`. An earlier version of this doc claimed the opposite, based on an older SDK or misread.
2. **"SDK lazily drains events during a print loop; fix must be upstream."** Wrong. The SDK buffers between `stream()` calls, which is fine — any consumer that keeps iterating picks them up. The bug is entirely on the host side.
3. **"Approach #1 (persistent consumer) broke Isomux when tried previously."** The older doc said this; it was attempted against an older SDK where per-turn stream semantics may have differed, and the analysis was wrong. With the V2 API as currently shipped (verified on `0.2.116`), the persistent-consumer architecture is straightforward and correct.
4. **"We need timestamp fields on assistant/result messages so the UI can re-sort."** Unnecessary. Once the consumer runs continuously, `task_notification` is processed before the next user turn, so ordering is correct at the log level with no UI-side reordering needed.

## Key V2 invariants (for reference)

- `unstable_v2_createSession().stream()` returns an `AsyncGenerator<SDKMessage, void>` that ends at the turn's `result`.
- Calling `stream()` again on the same session returns a fresh generator for the next turn's events.
- A fresh `stream()` with no buffered events blocks until events arrive; with buffered events, yields them immediately.
- An active `stream()` generator receives responses to `send()` calls made while it's blocked — no re-call needed.
- `session.close()` unblocks a parked `stream()` iterator.
